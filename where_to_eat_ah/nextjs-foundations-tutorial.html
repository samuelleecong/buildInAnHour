üìö Introduction

Welcome to the comprehensive Next.js 15 tutorial! This guide is designed to take you from zero to confident in building modern web applications with Next.js.

What is Next.js?

Next.js is a React framework that provides a complete solution for building production-ready web applications. It handles routing, server-side rendering, optimization, and much more out of the box.

Key Features

üöÄ Server-Side Rendering (SSR)

Generate HTML on the server for faster initial page loads and better SEO.

üìÅ File-Based Routing

Create routes by simply adding files to your project structure.

‚ö° Automatic Code Splitting

Only load the JavaScript needed for each page.

üîÑ API Routes

Build your backend API alongside your frontend.

ü§î Why Choose Next.js?

The Problem with Plain React

Plain React (Create React App) is a client-side only solution. This means:

  • ‚ùå Slower initial page loads (users see blank screen while JS downloads)
  • ‚ùå Poor SEO (search engines struggle with client-rendered content)
  • ‚ùå No built-in routing solution
  • ‚ùå Manual setup for optimization (code splitting, image optimization, etc.)
  • ‚ùå No easy way to mix server and client rendering

How Next.js Solves This

  • ‚úÖ Server-Side Rendering: HTML generated on server = instant content
  • ‚úÖ Static Site Generation: Pre-render pages at build time
  • ‚úÖ Built-in Routing: File-based routing system
  • ‚úÖ Automatic Optimization: Images, fonts, scripts optimized automatically
  • ‚úÖ Hybrid Approach: Mix server and client components seamlessly
  • ‚úÖ Server Actions: Call server functions directly from components
üí° Pro Tip: Next.js is production-ready out of the box. Companies like Netflix, TikTok, Twitch, and Nike use it in production.

üó∫Ô∏è The App Router (Next.js 13+)

‚ö†Ô∏è Important: Two Router Systems

Next.js has two routing systems:

  • Pages Router (old) - uses pages/ directory
  • App Router (new, Next.js 13+) - uses app/ directory

Use App Router for all new projects! It's the future of Next.js.

üîÑ Server Components vs Client Components

The Paradigm Shift

Next.js 13+ introduces React Server Components (RSC). This is a fundamental change in how we think about React:

By default, ALL components in the App Router are Server Components.

Server Components (Default)

‚úÖ What are Server Components?

Components that render on the server. They don't send JavaScript to the client.

Benefits:

  • Direct database access
  • Keep sensitive data on server (API keys, tokens)
  • Smaller JavaScript bundles
  • Better performance
  • Better SEO

Can use:

  • ‚úÖ async/await directly in component
  • ‚úÖ Backend libraries (databases, file systems)
  • ‚úÖ Environment variables

Cannot use:

  • ‚ùå useState, useEffect, other hooks
  • ‚ùå Browser APIs (window, document)
  • ‚ùå Event handlers (onClick, onChange)
// Server Component (default) - NO "use client" directive
import { db } from '@/lib/database'

export default async function UsersPage() {
  // Direct database access - runs on server only
  const users = await db.query('SELECT * FROM users')

  return (
    <div>
      <h1>Users</h1>
      <ul>
        {users.map(user => (
          <li key={user.id}>{user.name}</li>
        ))}
      </ul>
    </div>
  )
}

Client Components

‚ö†Ô∏è What are Client Components?

Components that render on the client (browser). They're like traditional React components.

Add "use client" directive at the top of the file to make it a Client Component.

Use when you need:

  • Interactivity (clicks, form inputs)
  • State management (useState)
  • Effects (useEffect)
  • Browser APIs
  • Event listeners
  • Custom hooks
"use client" // Makes this a Client Component

import { useState } from 'react'

export default function Counter() {
  const [count, setCount] = useState(0)

  return (
    <div>
      <p>Count: {count}</p>
      <button onClick={() => setCount(count + 1)}>
        Increment
      </button>
    </div>
  )
}

The Golden Rule

üéØ When to Use Each

Use Server Components by default. Only use Client Components when you need interactivity.

Think: "Can this component be static HTML?" If yes ‚Üí Server Component. If you need clicks, state, etc. ‚Üí Client Component.

Composition Pattern

// app/page.tsx - Server Component
import { ClientSidebar } from '@/components/client-sidebar'
import { db } from '@/lib/database'

export default async function Page() {
  const data = await db.getData()

  return (
    <div>
      {/* Server-rendered content */}
      <h1>{data.title}</h1>

      {/* Client component for interactivity */}
      <ClientSidebar items={data.items} />
    </div>
  )
}

// components/client-sidebar.tsx - Client Component
"use client"

import { useState } from 'react'

export function ClientSidebar({ items }) {
  const [isOpen, setIsOpen] = useState(false)

  return (
    <aside>
      <button onClick={() => setIsOpen(!isOpen)}>
        Toggle
      </button>
      {isOpen && <ul>{/* items */}</ul>}
    </aside>
  )
}

üìä Data Fetching

The New Way

In Next.js 15 with App Router, data fetching is drastically simplified. You can use async/await directly in Server Components!

Server Component Data Fetching

// app/posts/page.tsx - Server Component
export default async function PostsPage() {
  // Fetch data directly in the component
  const res = await fetch('https://api.example.com/posts')
  const posts = await res.json()

  return (
    <div>
      {posts.map(post => (
        <article key={post.id}>
          <h2>{post.title}</h2>
          <p>{post.excerpt}</p>
        </article>
      ))}
    </div>
  )
}

Caching Strategies

Next.js automatically caches fetch requests. You can control this behavior:

1. Static Data (Default)

// Cached indefinitely (until next build)
const res = await fetch('https://api.example.com/posts')

// Equivalent to:
const res = await fetch('https://api.example.com/posts', {
  cache: 'force-cache'
})

2. Dynamic Data

// Never cache - always fetch fresh
const res = await fetch('https://api.example.com/posts', {
  cache: 'no-store'
})

3. Revalidate (ISR - Incremental Static Regeneration)

// Revalidate every 60 seconds
const res = await fetch('https://api.example.com/posts', {
  next: { revalidate: 60 }
})

Database Queries

// app/users/page.tsx
import { db } from '@/lib/database'

export default async function UsersPage() {
  // Direct database access in Server Component
  const users = await db.user.findMany()

  return (
    <ul>
      {users.map(user => (
        <li key={user.id}>{user.name}</li>
      ))}
    </ul>
  )
}

Parallel Data Fetching

export default async function Dashboard() {
  // Fetch in parallel for better performance
  const [users, posts, comments] = await Promise.all([
    fetch('https://api.example.com/users').then(r => r.json()),
    fetch('https://api.example.com/posts').then(r => r.json()),
    fetch('https://api.example.com/comments').then(r => r.json()),
  ])

  return (
    <div>
      <UserList users={users} />
      <PostList posts={posts} />
      <CommentList comments={comments} />
    </div>
  )
}

‚ö° Server Actions

What are Server Actions?

Server Actions let you call server-side functions directly from your components, without creating API routes. They're perfect for forms, mutations, and any server-side logic.

Basic Server Action

// app/actions.ts
'use server'

export async function createPost(formData: FormData) {
  const title = formData.get('title')
  const content = formData.get('content')

  // Save to database
  await db.post.create({
    data: { title, content }
  })

  // Redirect after success
  redirect('/posts')
}

Using Server Actions in Forms

// app/new-post/page.tsx
import { createPost } from '../actions'

export default function NewPostPage() {
  return (
    <form action={createPost}>
      <input name="title" placeholder="Title" required />
      <textarea name="content" placeholder="Content" required />
      <button type="submit">Create Post</button>
    </form>
  )
}

With Client Components (using useFormState)

'use client'

import { useFormState } from 'react-dom'
import { createPost } from '../actions'

export function CreatePostForm() {
  const [state, formAction] = useFormState(createPost, null)

  return (
    <form action={formAction}>
      <input name="title" />
      <textarea name="content" />
      <button type="submit">Submit</button>
      {state?.error && <p>{state.error}</p>}
    </form>
  )
}

Non-Form Server Actions

'use client'

import { deletePost } from '../actions'

export function DeleteButton({ postId }: { postId: string }) {
  const handleDelete = async () => {
    await deletePost(postId)
  }

  return (
    <button onClick={handleDelete}>
      Delete Post
    </button>
  )
}

Benefits of Server Actions

  • No need to create API routes
  • Type-safe (with TypeScript)
  • Automatic revalidation of data
  • Works with or without JavaScript enabled
  • Built-in security (CSRF protection)
üí° Pro Tip: Server Actions are the recommended way to handle mutations in Next.js 15. They replace the need for most API routes.

Security Best Practices

1. Environment Variables

// .env.local (never commit this file!)
DATABASE_URL="postgresql://..."
API_SECRET_KEY="secret123"

// Access in Server Components only
const secret = process.env.API_SECRET_KEY

// For client-side (must prefix with NEXT_PUBLIC_)
NEXT_PUBLIC_API_URL="https://api.example.com"

2. Validate User Input

'use server'

import { z } from 'zod'

const PostSchema = z.object({
  title: z.string().min(1).max(100),
  content: z.string().min(1),
})

export async function createPost(formData: FormData) {
  // Validate input
  const result = PostSchema.safeParse({
    title: formData.get('title'),
    content: formData.get('content'),
  })

  if (!result.success) {
    return { error: 'Invalid input' }
  }

  // Safe to use
  const { title, content } = result.data
  // ...
}

Naming Conventions

  • Components: PascalCase (e.g., UserProfile.tsx)
  • Utilities: camelCase (e.g., formatDate.ts)
  • Constants: UPPER_SNAKE_CASE (e.g., MAX_ITEMS)
  • Folders: kebab-case (e.g., user-profile/)
  • Event handlers: handleEventName (e.g., handleClick)

Common Pitfalls to Avoid

  • ‚ùå Don't use useState in Server Components
  • ‚ùå Don't fetch data in Client Components (use Server Components)
  • ‚ùå Don't put "use client" at the top of every file
  • ‚ùå Don't use <a> for internal links (use <Link>)
  • ‚ùå Don't import Server Components into Client Components
  • ‚ùå Don't forget to add alt text to images
  • ‚ùå Don't commit .env.local files

Quick Reference Checklist
